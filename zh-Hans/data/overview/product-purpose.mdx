---
title: 产品实现目的
description: UOSE系统要解决的业务问题、用户角色与成功标准。
---

UOSE系统的实现目的，是让企业能够把分散的业务系统、数据资产和知识资产安全地交给智能体使用。它不是把所有系统重建一遍，而是在现有系统之上建立统一语义控制面和可治理执行面。

## 面向的用户角色

UOSE系统服务四类角色：

- 数据平台管理员：负责资源注册、连接密钥、同步任务和运行状态。
- 业务建模人员：关注指标、模型、数据库表、知识实体等对象是否语义清晰。
- 智能体构建者：把业务 Assistant 绑定到可访问资源，并通过 MCP 或 ChatKit 使用 UOSE 工具。
- 治理与审计人员：配置策略、处理审批、查看执行审计和异常事件。

## 产品目标

UOSE系统要达成以下目标：

1. 让外部资源可见：把资源、模型、服务、表、字段、知识实体统一登记。
2. 让业务语义可读：把对象关系、属性、别名、约束和证据发布为本体快照。
3. 让智能体可用：提供标准 Agent Tools，而不是暴露任意后端接口。
4. 让执行可控：通过 action manifest、策略、模拟和审批保护执行边界。
5. 让过程可追溯：记录同步、动作、审批和执行审计，支持复盘。

## 从数据目录到操作系统

传统数据目录更多解决“资产在哪里”和“字段是什么”。UOSE系统进一步解决“对象能做什么”“Agent 是否能做”“执行结果如何证明”。

这意味着 UOSE 的资产不是静态资产，而是可操作资产：

- 指标不仅可被浏览，还可以被查询趋势。
- Cube 不仅可被查看，还可以做切片分析。
- SAP Entity Set 不仅可被识别，还可以读取集合、读取实体、创建或更新。
- 数据库表不仅可被列出，还可以描述、预览、解释和执行只读查询。
- 知识实体不仅可被检索，还可以沿图谱关系查看证据。

## 成功标准

一个 UOSE 实施可以用以下标准衡量：

- 资源接入成功率高，失败能够进入可排查的 dead-letter 或 sync job 状态。
- 当前组织下的资源有可用 ontology snapshot。
- Agent 能通过固定流程定位对象、获取邻域、发现动作并执行低风险动作。
- 高风险动作能被策略识别，并进入审批或被拒绝。
- 执行结果包含足够审计信息，可追踪到任务、资源、动作、目标和策略判定。
- 新资源类型能通过 adapter manifest 与 capabilities schema 扩展，而不破坏现有 Agent 工具协议。

## 设计取舍

UOSE系统优先选择稳定和可治理：

- 优先结构化契约，而不是让 Agent 自由解释文档。
- 优先资源级接入，而不是无边界跨系统调用。
- 优先模拟和审计，而不是直接执行。
- 优先渐进式上下文，而不是一次性暴露全量元数据。
- 优先可扩展 adapter，而不是为每个系统写一次性逻辑。

这些取舍让 UOSE 更适合企业级智能体生产环境。
